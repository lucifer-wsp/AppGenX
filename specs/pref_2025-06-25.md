> 生成日期：2025-06-25  
> 用途：供新 Agent / 开发者快速理解项目全貌、维护现有功能、按规范扩展新能力。

---

## 1. 项目定位

**AppGen** 是从 App Store 商机发现到 iOS 上架物料生成的**全链路多智能体系统**。

核心能力：

1. **市场扫描（Scan）**：并发拉取美区/欧洲 App Store 榜单 → LLM 分析 Micro-App 机会
2. **产品流水线（Pipeline）**：10 个 Agent 依次产出 PRD、设计、SwiftUI 工程、编译验证、测试与上架文案
3. **人机 Review 门禁**：每阶段可 CLI / Web / 自动 三种模式审批
4. **Web Dashboard**：Vue 3 + Naive UI，扫描配置、机会选择、流水线进度与文档审阅

## 4. 流水线架构

### 4.1 阶段顺序

```
Scout → Analyst → PM → Designer → DevInit → DevScaffold → DevCode → DevVerify → QA → Store
         ↑ 每阶段完成后触发 ReviewManager 门禁
```

定义于 `appgen/pipeline.py` 的 `PipelineOrchestrator.STAGE_ORDER`。

### 4.2 各 Agent 职责与产物

| Agent | 模块 | 输出模型 | 主要文件 |
|-------|------|----------|----------|
| Scout | `agents/scout.py` | `OpportunityBrief` | `01_opportunity.json` |
| Analyst | `agents/analyst.py` | `RequirementSpec` | `02_requirements.json` |
| PM | `agents/pm.py` | `PRDDocument` | `03_prd.md` |
| Designer | `agents/designer.py` | `DesignSpec` | `04_design.md` |
| DevInit | `agents/dev_init.py` | `DevInitPlan` | `05_dev_plan.json` |
| DevScaffold | `agents/dev_scaffold.py` | — | `project/project.yml`, Swift 骨架 |
| DevCode | `agents/dev_code.py` | `DevCodeManifest` | `05_dev_manifest.json`, `project/**/*.swift` |
| DevVerify | `agents/dev_verify.py` | `BuildReport` | `05_build_report.json` |
| QA | `agents/qa.py` | `TestPlan` | `06_test_plan.md` |
| Store | `agents/store.py` | `StoreListing` | `07_store_listing.md`, `fastlane/metadata/` |

### 4.3 Review 系统

- **模式**：`cli`（终端交互，默认）/ `auto`（自动通过）/ `web`（Dashboard 暂停）
- **单次运行 override**：`run.metadata.auto_review`（CLI `--auto`、Web 顶栏开关）
- **状态**：`pending` → `approved` / `rejected` / `revision_requested`
- **revision_requested**：重跑当前 Agent 后再 Review
- **非 approved**：流水线 `status = "paused"`，可 `appgen resume <run_id>`

Review 质量门槛见 `docs/PLAYBOOKS.md`（如 Scout 置信度 ≥ 60、Analyst MVP ≤ 5 项）。

### 4.4 扩展新 Agent 步骤

1. 在 `appgen/models/` 定义 Pydantic 输出模型
2. 继承 `BaseAgent`，实现 `async run()`
3. 注册到 `PipelineOrchestrator.STAGE_ORDER` 与 `_agents`
4. 在 `review.py` 的 `_stage_summary` 添加摘要
5. 如需 Markdown 输出，在 `appgen/templates/` 添加 Jinja2 模板

## 5. 市场扫描（Scan）

### 5.1 扫描模式

| 模式 | 触发条件 | 说明 |
|------|----------|------|
| `keyword` | 有关键词、无品类 | 关键词搜索 + 各区域 Top N 榜单 |
| `full` | 无关键词、无品类 | 全部 24 品类 × 区域 × 榜单 |
| `categories` | 指定 genre_ids | 选定品类子集 |

计算逻辑：`appgen/scan_flow.py` → `plan_scan()`

### 5.2 两阶段执行

**Phase 1 — fetch（拉榜）**

- 并发队列：`primary` 队列 + `retry` 队列
- **失败处理（2025-06-25 更新）**：
  - 可重试错误（403/429 限流、超时、5xx、连接失败）→ 加入 **retry 队列队尾**，先继续处理其他 item
  - 一轮 primary 全部完成后，对 retry 队列做指数退避，再进入下一轮
  - 非可重试错误（legacy RSS 403、空结果）→ 永久失败，不重试
  - 最多队尾重试 `SCAN_FETCH_MAX_JOB_RETRIES = 5` 次
- 并发治理：`ScanFetchConcurrencyGovernor`（10→6→2→1 阶梯，429 降档，连续 3 轮成功升档）
- HTTP 层：`AppStoreClient._get_json` 最多 5 次即时重试（与队列层互补）

**Phase 2 — analyze（LLM 分析）**

- 快照采样上限：`analyze_max_snapshots`（默认 40，Web 可配 10–120）
- 分批：`analyze_batch_size=6`，并发 `analyze_concurrency=3`
- 机会**增量写入** `scan_live.json`，完成后按 `confidence_score` 重排
- 单批失败：记日志、继续其他批；全部失败才抛错
- Cursor CLI 模式：错峰启动（stagger + jitter）

### 5.3 扫描状态

```
status: running | completed | failed | cancelled
phase:  fetch | analyze | done
```

**取消扫描**（2025-06-25 更新）：

- 后端：`POST /api/scans/{id}/cancel` → `scan_cancel.request(scan_id)`
- 拉榜循环与 LLM 分析循环内检查取消信号
- 取消后保留已采集快照与已分析机会，`status = cancelled`
- 前端：扫描状态 panel 显示「停止扫描」（进行中可见）；机会 panel 显示「停止分析」（LLM 阶段可见）

**重新分析**：`POST /api/scans/{id}/reanalyze` — 复用已有榜单，仅重做 LLM

### 5.4 扫描产物

```
workspace/scans/<scan_id>/
├── scan.json              # 完整状态
├── scan_live.json         # 轻量实时状态（轮询用）
├── categories/            # 各品类原始 JSON
├── opportunities.json     # 结构化机会
└── scan_report.md         # 可读报告
```

### 5.5 区域与榜单

- **覆盖**：美国 + 欧洲 10 国（英/德/法/西/意/荷/瑞/波/爱/葡）
- **屏蔽**：`cn`, `hk`, `mo`, `tw`
- **预设**：`us` / `eu` / `us-eu`（默认）
- **榜单**：`top-free`, `top-paid`（marketing API）；`top-grossing` 品类 RSS 已关闭，自动跳过

### 5.6 机会反馈偏好

- 存储：`workspace/scan_preferences.json`
- Web 标记 `good` / `bad` + 可选 reason
- 后续扫描/重分析时注入 LLM prompt，避开 bad、参考 good

### 5.7 进入流水线

```bash
appgen scan pick <scan_id> <rank> [--auto]
# 或 POST /api/scans/{id}/pick
```

写入 `run.metadata.picked_from_scan` / `picked_opportunity`，Scout 走 `scan_bridge` 跳过重新挖掘。

## 8. 开发规范

### 8.1 Python 代码规范

- **行宽**：100 字符（ruff）
- **常量**：集中在 `appgen/constants.py`，避免魔法数字/字符串
- **类型**：Pydantic v2 模型 + type hints
- **异步**：扫描/流水线核心路径用 asyncio；LLM 同步调用包在 `asyncio.to_thread`
- **错误**：用户面向错误用 `scan_errors.format_scan_error()`；日志用 `scan.log()` + `_flush()`
- **最小 diff**：只改需求相关代码，匹配现有命名与抽象风格
- **注释**：仅解释非显而易见的业务/技术细节

### 8.2 iOS 代码规范（DevCode）

定义于 `agents/dev_code.py` → `PRODUCTION_CODING_RULES`：

1. SwiftUI + MVVM（@Observable ViewModel）
2. 禁止 force unwrap
3. 颜色/字体用 AppTheme
4. 遵循 DesignSpec
5. 完整空态/错误态/加载态
6. accessibilityLabel
7. 禁止 placeholder TODO UI

### 8.3 前端规范

- Vue 3 Composition API（`<script setup>`）
- Naive UI 组件，暗色主题（Soybean Admin 风格）
- API 调用统一走 `frontend/src/api/client.js`
- 状态持久化用 sessionStorage（非 localStorage）
- 组件 scoped CSS，BEM 风格 class 名

### 8.4 测试规范

```bash
pytest                          # 全部测试
ruff check appgen                 # 静态检查
```

- 框架：pytest + pytest-asyncio（`asyncio_mode = auto`）
- Mock：测试 fixture 设 `LLM_PROVIDER=mock`、空 Key、`APPGEN_REVIEW_MODE=auto`
- 隔离：`tmp_path` 作 `APPGEN_WORKSPACE`
- 扫描队列测试：`tests/test_scan_fetch_queue.py`
- 新增功能应补充对应测试，覆盖真实行为而非 trivial assert

### 8.5 Git / 提交

- 仅在用户明确要求时 commit
- 不修改 git config，不 force push
- commit message 聚焦「为什么」

## 10. 常见维护场景
